## Бизнес-проблема

Сейчас клиент не может самостоятельно подать заявку на досрочное расторжение договора в личном кабинете.

Для подачи заявки клиент вынужден обращаться в офис, контакт-центр или поддержку. Это увеличивает нагрузку на сотрудников и ухудшает клиентский опыт.

## Бизнес-цель

Дать клиенту возможность самостоятельно подать заявку на досрочное расторжение договора через личный кабинет.

## Ожидаемый бизнес-результат

Клиент сможет:

- выбрать договор в личном кабинете;
- заполнить данные для расторжения;
- отправить заявку;
- получить номер заявки;
- отслеживать статус заявки.

Компания сможет:

- снизить нагрузку на контакт-центр и офисы;
- стандартизировать сбор данных для расторжения;
- уменьшить количество ошибок при ручном оформлении;
- повысить прозрачность процесса для клиента.

## Бизнес-требования

### BR.01. Подача заявки через личный кабинет

Система должна позволять клиенту подать заявку на досрочное расторжение договора из карточки договора в личном кабинете.

### BR.02. Доступность операции только для активных договоров

Операция досрочного расторжения должна быть доступна только для договоров в статусе `ACTIVE`.

### BR.03. Запрет дублей

Система не должна позволять создать новую заявку на расторжение, если по договору уже есть активная заявка в одном из статусов:

- `DRAFT`;
- `SUBMITTED`;
- `IN_REVIEW`;
- `WAITING_FOR_DOCUMENTS`;
- `APPROVED`.

### BR.04. Сбор обязательных данных

При создании заявки клиент должен указать:

- причину расторжения;
- желаемую дату расторжения;
- банковские реквизиты для возврата денежных средств.

### BR.05. Документы

Система должна позволять клиенту приложить документы к заявке, если они требуются для выбранной причины расторжения.

### BR.06. Номер заявки

После успешной подачи система должна создать уникальный номер заявки и показать его клиенту.

### BR.07. Уведомление клиента

После создания заявки клиент должен получить уведомление о принятии заявки в работу.

### BR.08. Отслеживание статуса

Клиент должен иметь возможность видеть статус заявки в личном кабинете.

## Бизнес-ограничения

| Ограничение | Описание |
|---|---|
| Только авторизованные клиенты | Подать заявку может только авторизованный пользователь |
| Только свой договор | Клиент может создать заявку только по договору, который принадлежит ему |
| Только активный договор | По завершенным, аннулированным или уже расторгнутым договорам заявка не создается |
| Одна активная заявка | По одному договору может быть только одна активная заявка на расторжение |

## Критерии приемки

| ID | Критерий |
|---|---|
| AC.01 | Клиент может открыть форму расторжения из карточки активного договора |
| AC.02 | Клиент может заполнить обязательные поля и отправить заявку |
| AC.03 | После успешной отправки система возвращает номер заявки |
| AC.04 | Если договор не активен, операция недоступна |
| AC.05 | Если по договору уже есть активная заявка, новая заявка не создается |
| AC.06 | Если обязательные поля не заполнены, система возвращает ошибку валидации |
| AC.07 | После создания заявки клиент получает уведомление |
| AC.08 | Заявка отображается в списке заявок клиента |

## Не входит в объем задачи

| Область | Причина исключения |
|---|---|
| Расчет суммы возврата | Выполняется отдельным backend-процессом после создания заявки |
| Согласование заявки сотрудником | Описывается отдельной задачей операционного контура |
| Подписание заявления электронной подписью | Может быть реализовано отдельным этапом |
| Интеграция с платежной системой | Не требуется для создания заявки |
